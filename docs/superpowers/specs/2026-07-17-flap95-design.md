## 2. Game loop & economy

### Core loop (one tap)
- A bird flies through gaps between pipes; tap to flap; collision ends the run.
- Each run < 60 seconds; instant retry on death. Difficulty ramps with score.
- **Deterministic** engine: the world is generated from a `seed`; same seed +
  same tap-timing sequence ⇒ exactly the same outcome (prerequisite for ghosts
  and anti-cheat).

### Modes
1. **Practice (free):** unlimited, fully off-chain. Off-chain high-score
   leaderboard for retention.
2. **1v1 duels (asynchronous):**
   - The creator picks a stake of **0.1 / 0.5 / 1 USDm**, approves + deposits
     into the escrow contract, then plays one run with a backend-issued seed.
     The replay (tap trace) is stored; the duel is listed as an "open duel".
   - The acceptor matches the stake and plays the **same seed**, with the
     **creator's ghost flying alongside**. Higher score wins the pot minus a
     **5% fee** to the treasury.
   - Tie: both stakes refunded (including a 0–0 tie).
   - No acceptor within **24h**: the creator can withdraw (100% refund).
   - **"Rematch" button** after a loss: creates a new duel flagged as a
     challenge to that specific opponent (shown to them with priority; anyone
     else can accept it after 1h without a response).
3. **Cold start:** pre-seed 5–10 "house" duels from real runs recorded by the
   dev (dev wallet, real stakes, transparently labeled as house ghosts in the
   UI) so day-one visitors always have a duel to accept.

### Currency & gas
- Stakes in **USDm (cUSD)** — mainnet `0x765DE816845861e75A25fCA122bb6898B8B1282a`
  (18 decimals).
- Gas: MiniPay wallets hold no CELO ⇒ all transaction guidance uses
  `feeCurrency` = the USDm adapter (look up via `FeeCurrencyDirectory`
  `0x15F344b9E6c3Cb6F0376A36A64928b13F62C6276` at implementation time; MiniPay's
  injected provider handles most of this automatically).

## 3. Anti-cheat

- The client **never submits a score** — only `(duelId, seed, tapTimestamps[])`.
- The backend re-runs the exact deterministic engine (same engine source,
  running on Node) to compute the score from the tap trace. The client-side
  score is only a local prediction.
- The backend signs results with an **oracle key** (EIP-712: `duelId, winner,
  scoreA, scoreB`). The contract's `settle()` accepts only oracle signatures.
- Sanity limits on traces: max taps/second (blocks superhuman spam bots), max
  run length. Violations ⇒ invalid run, scored as a loss.
- Accepted residual risk (documented, not addressed in MVP): a player writing a
  skilled bot. With stakes ≤ 1 USDm, attack cost exceeds the payoff.

## 4. Architecture

```
MiniPay/Browser ──> Next.js (Vercel)
  │ Canvas game engine (deterministic, plain TS)
  │ wagmi/viem + MiniPay injected provider
  ├─ API routes: issue seed, receive trace, verify + oracle-sign, list duels
  ├─ Postgres (Neon via Vercel Marketplace): duels, replays, free leaderboard
  └─ DuelEscrow.sol (Celo Mainnet, Foundry)
       createDuel / acceptDuel / settle(sig) / cancelExpired
```

### `DuelEscrow` contract (single contract)
- `createDuel(stake)` — transferFrom USDm into escrow, emit `DuelCreated(id)`.
  Stake must be one of the 3 tiers (blocks junk duels).
- `acceptDuel(id)` — locks in the acceptor + matching stake; reverts if the
  duel has expired.
- `settle(id, winner, scoreA, scoreB, oracleSig)` — verifies EIP-712, pays
  `2*stake*95%` to the winner, 5% to the treasury; on a tie refunds both.
  Callable by anyone (typically the backend relayer or the winner claiming).
- `cancelExpired(id)` — after 24h with no acceptor, refunds the creator.
- Minimal admin surface: `oracle`, `treasury`, `owner` (rotate the oracle if the
  key leaks). Not upgradeable, no elaborate pause machinery — keep the contract
  small and hand-auditable.

### Duel data flow (happy path)
1. FE calls `POST /api/duels` ⇒ backend generates a seed, stores a draft duel.
2. FE walks the user through USDm approve + `createDuel(stake)` ⇒ backend
   reconciles against the event; duel transitions to `open`.
3. Creator plays ⇒ `POST /api/duels/:id/replay` (tap trace) ⇒ backend verifies,
   stores score A + the trace as the ghost.
4. Acceptor calls `acceptDuel(id)` ⇒ plays the same seed against ghost A ⇒
   submits trace ⇒ backend verifies, determines the winner, signs EIP-712,
   calls `settle()`.
5. FE shows the result (Win95 dialog) + Rematch button.

### Key error handling
- **Deposited but never played / never submitted a trace:** 1h after accepting,
  a side with no valid trace is signed as a forfeit (score 0). A creator who
  never plays ⇒ the duel never reaches `open` and falls through to
  `cancelExpired`.
- **Backend/oracle down:** no funds are lost — unsettled duels stay in escrow;
  worst case players wait for the oracle to come back; `cancelExpired` is the
  exit for unaccepted duels. (Accepted: an accepted duel with a permanently
  dead oracle is stuck — noted as a future upgrade if needed.)
- **Reorgs/failed txs:** the FE always reads state from the backend (which
  indexes events) and shows pending states clearly with a Win95 progress bar.

## 5. UI/UX — Win95/98 visual language, mobile skeleton

- Foundation: `98.css` (a few dozen KB) + customization; pixel font; teal
  desktop background.
- **Mobile-first skeleton:** single column, tap targets ≥ 44px, every primary
  action within thumb reach; title bars / X buttons are decorative, never the
  primary navigation.
- Screen mapping:
  - Home = a "desktop" with icons: Play, Open Duels, Hall of Fame, Wallet.
  - Duel list = a File Explorer-style window, one "file" row per duel.
  - Losing a duel = classic error dialog: ⚠️ "You lost 0.2 USDm — [Rematch]
    [Close]".
  - Pending transaction = segmented blue progress bar + hourglass cursor.
  - Leaderboard = an Excel 97-style grid.
  - Game canvas: clean, framed in a title-barred window; consistent pixel art.
- **Language:** English only. Careful copywriting (an explicit review
  criterion).
- `isMiniPay()` hook: inside MiniPay hide external wallet-connect flows and use
  the injected provider; outside MiniPay free play still works and a standard
  wallet connect is available.

## 6. Testing & acceptance

- **Contract (Foundry):** create/accept/settle across win-lose-tie, invalid
  signature must revert, cancelExpired honors the deadline, invalid stake tier
  reverts, fee routes to the treasury. Basic fuzzing on stake/score.
- **Engine:** deterministic golden tests — fixed (seed, trace) pairs ⇒ fixed
  scores, matching across browser and Node; taps-per-second limit tests.
- **Manual E2E before submission:** full create → accept → settle flow on the
  MiniPay site tester with 2 real wallets, on mainnet, at the 0.1 USDm tier.
- **Bundle:** verify `next build` first-load JS < 2MB.
